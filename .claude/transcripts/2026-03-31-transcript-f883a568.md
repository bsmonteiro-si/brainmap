Here's the full picture:

## CLAUDE.md Hierarchy

**Three scopes**, all loaded into context (they accumulate, not replace):

| Scope | Location | Purpose |
|-------|----------|---------|
| **Managed/Enterprise** | `/Library/Application Support/ClaudeCode/CLAUDE.md` (macOS) | Org-wide policy, can't be excluded |
| **User** | `~/.claude/CLAUDE.md` | Personal prefs across all projects |
| **Project** | `./CLAUDE.md` or `./.claude/CLAUDE.md` | Team instructions, checked into git |

**Directory tree walk**: Claude Code walks **up** from your cwd, loading every CLAUDE.md it finds in ancestor directories. Subdirectory CLAUDE.md files load lazily (on-demand when Claude reads files there).

**Precedence**: Managed > Project > User (more specific overrides broader).

## Path-Scoped Rules

For granularity, `.claude/rules/*.md` files can have `paths` frontmatter — they only load when Claude touches matching files:

```markdown
---
paths: ["src/**/*.tsx"]
---
# React-specific rules here
```

Rules without `paths` load at startup like the main CLAUDE.md.

## settings.json Hierarchy

Four tiers, highest priority first:

1. **Managed** — `/Library/Application Support/ClaudeCode/managed-settings.json`
2. **Local project** — `.claude/settings.local.json` (git-ignored)
3. **Shared project** — `.claude/settings.json` (committed)
4. **User** — `~/.claude/settings.json`

Arrays (like `permissions.allow`) **merge** across scopes. Scalars are replaced by the higher-priority scope.
