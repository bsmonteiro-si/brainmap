## Assistant

Looking at this canvas, the structure is actually a **decision flow**, not just a pipeline — and the visual representation doesn't fully capture that. Here are some patterns worth discussing:

---

**1. Make the decision nature explicit**

Right now, do-calculus branches into 3 outputs with no indication of *why* each path is taken. A decision diamond between do-calculus and the strategies would communicate that the user is choosing a strategy based on conditions:

```
do-calculus → ◇ "Which case?" → [Back-Door] if confounders observable
                              → [Front-Door] if mediator exists
                              → [Instrumental Var.] otherwise
```

The condition labels currently live inside callout nodes. They belong on the edges instead.

---

**2. Callout as annotation vs. callout as description card**

Right now callouts sit *above* the concept and explain it in advance. An alternative is inverting the hierarchy: the callout IS the card (bigger, visually primary), and the file/concept node is a smaller "link" pinned below it. The callout becomes the reading surface, the file node becomes the navigation target.

---

**3. A single "Strategies" group with a swimlane feel**

Instead of three separate groups for Back-Door / Front-Door / IV, one "Strategies" group containing all three — with the three concepts as rows — reads more naturally as "these are options within the same category."

---

**4. Semantic color coding**

Use colors to encode role, not just grouping:
- Blue → input/prerequisite (Causal Discovery)
- Neutral → core tool (do-calculus)
- Red/orange → problem/hazard (Confounding)
- Green → strategy/solution (Back-Door, Front-Door, IV)

---

**5. Drop the groups, embrace edge labels as annotations**

Groups add visual clutter. An alternative: remove groups entirely, move all descriptions into **edge labels** or **inline node text** (larger cards that include the description directly in the node body). Cleaner, flatter.

---
