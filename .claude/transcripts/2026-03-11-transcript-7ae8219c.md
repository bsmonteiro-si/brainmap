# Fix: Opening arbitrary folder loads ancestor workspace

## Context

`Workspace::open()` calls `find_workspace_root()` which walks UP the directory tree to find `.brainmap`. When the desktop app opens `/Users/bsmonteiro/Desktop/LaTex`, this finds `.brainmap` at the brainmap project root and loads its seed data (34 nodes, 116 edges).

The previous fix (auto-init before open) didn't work because the workspace store's `openWorkspace` catches errors internally and never throws — so even if init fails, the frontend may still display stale/wrong data. We need a more robust approach.

## Fix: Two changes

### 1. Add `Workspace::open_or_init` to core (`crates/core/src/workspace.rs`)

New public method that does NOT walk up the directory tree:
- If `path/.brainmap` exists → load workspace from that exact path (same logic as `open` but using `path` directly instead of `find_workspace_root`)
- If `path/.brainmap` doesn't exist → call `init(path)` first, then load

This is distinct from `open()` which preserves its walk-up behavior for CLI/MCP use.

```rust
/// Open a workspace rooted exactly at `path`, initializing if needed.
/// Unlike `open()`, this does NOT walk up the directory tree.
pub fn open_or_init(path: &Path) -> Result<Self> {
    let brainmap_dir = path.join(BRAINMAP_DIR);
    if !brainmap_dir.is_dir() {
        Self::init(path)?;
    }
    // Load directly from this path — no find_workspace_root
    let mut config = load_config(&brainmap_dir)?;
    let md_files = scan_md_files(path);
    // ... same loading logic as open() but using `path` as root
}
```

To avoid duplicating the loading logic, extract the post-root-resolution code from `open()` into a private `load_from_root(root: &Path)` helper, then have both `open()` and `open_or_init()` call it.

### 2. Use `open_or_init` in Tauri handler (`crates/app/src-tauri/src/handlers.rs`)

Replace:
```rust
let ws = Workspace::open(p).map_err(...)?;
```
With:
```rust
let ws = Workspace::open_or_init(p).map_err(...)?;
```

Remove the separate init-before-open check we added previously.

### 3. Fix frontend: workspace store error propagation

The `handleOpenFolder` in SegmentPicker calls `openWorkspace(segment.path)` from the workspace store, which **never throws** (catches internally). This means `touchSegment` and `setActiveSegmentId` always run even on failure. Fix by checking `info` after the call:

```typescript
await openWorkspace(segment.path);
const info = useWorkspaceStore.getState().info;
if (!info) {
  // openWorkspace failed internally
  setCardErrors((prev) => ({ ...prev, [segment.id]: useWorkspaceStore.getState().error || "Failed to open" }));
  setOpeningId(null);
  return;
}
touchSegment(segment.id);
setActiveSegmentId(segment.id);
```
