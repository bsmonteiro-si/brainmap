## Step 0: Shared Core Additions

These methods are needed by both CLI and MCP. Add to `crates/core/src/workspace.rs` unless noted otherwise.

### New Workspace Methods

**`delete_link(&mut self, source: &str, target: &str, rel: &str) -> Result<()>`**
- Remove the TypedLink from source note's frontmatter where target resolves to the given path and rel matches
- Rewrite source file, update graph (remove_edge), update index

**`move_note(&mut self, old_path: &str, new_path: &str) -> Result<Vec<String>>`**
- Rename file on disk (create parent dirs for new path)
- Rewrite all notes that reference old_path (update their frontmatter links + inline links)
- Update graph (remove old node, add new node, rewire edges)
- Update index
- Return list of files that were rewritten (for display)

**`reindex(&mut self) -> Result<()>`**
- Rebuild index from current in-memory state (delegate to `index.rebuild()`)

**`find_path(&self, source: &str, target: &str, max_depth: Option<usize>) -> Result<Option<Vec<Edge>>>`**
- Thin wrapper around `graph.shortest_path()` with RelativePath conversion and FileNotFound validation

**`get_subgraph(&self, center: &str, depth: usize, rel_filter: Option<&str>) -> Result<Subgraph>`**
- Thin wrapper around `graph.subgraph()` with validation

### New Export Module: `crates/core/src/export.rs`

**`export_json(workspace: &Workspace) -> String`** — Full graph as JSON (nodes + edges)
**`export_dot(workspace: &Workspace) -> String`** — Graphviz DOT format
**`export_graphml(workspace: &Workspace) -> String`** — GraphML XML format

All three accept optional subgraph filter (center path + depth).

### Config Mutations in `crates/core/src/config.rs`

**`WorkspaceConfig::add_federation(&mut self, name: &str, path: &str)`**
**`WorkspaceConfig::remove_federation(&mut self, name: &str) -> bool`**

### Tools to Implement (18 for v1)

**Node (5):** node_get, node_list, node_create, node_update, node_delete

**Search (1):** search

Each tool:
- Receives JSON arguments via MCP protocol
- Calls the corresponding `Workspace` method
- Returns response envelope: `{"success": bool, "data": ..., "error": ...}`

Deferred tools (not in v1): config_set, federation_*, batch

### MCP Resources (3)

- `brainmap://nodes/{path}` — raw note content (calls `workspace.read_note`)
- `brainmap://graph` — full graph JSON (calls `export_json`)
- `brainmap://config` — workspace config YAML

### Server Lifecycle

The MCP server holds a `Workspace` instance and serves it over stdio:

```rust
struct BrainMapMcp {
    workspace: Workspace,
}
```

Started via: `brainmap serve --mcp-only --workspace <path>`

### Integration with CLI

Add `Serve` command to `crates/cli/src/main.rs`:
```rust
Serve {
    #[arg(long)]
    mcp_only: bool,
}
```

The serve command opens the workspace, creates `BrainMapMcp`, and starts the stdio server.

### Tests

- `crates/mcp/tests/` — integration tests that create a workspace, instantiate the server, and call tools programmatically (without going through stdio)
- Test each tool's success and error cases
