1. **The `index` note type exists** (e.g., "The Book of Why" is `type: index`) but has **zero special graph behavior** — it renders identically to every other node.
2. **Focus mode exists** and already shows a node + 1-hop neighbors as a clean subgraph.
3. **Folder nodes exist** as structural anchors.

The natural solution is to **combine these**: auto-focus the graph on the index note when opening a workspace, giving users a clean starting point instead of the full hairball.

Here's what I'm thinking:

### Core: Index-Anchored Graph Start
- On workspace open, detect if an `index`-type note exists → auto-apply focus mode on it
- Graph starts showing only the index + its direct neighbors (clean, navigable)
- User can clear focus (existing "Focus ×" button) to see the full graph

### Enhancements
- **Home button** in graph toolbar — one click returns to index-focused view
- **Visual distinction** for the index node — slightly larger, maybe a subtle glow or ring to signal "this is the hub"
- **Progressive exploration** — double-click already expands neighbors; this becomes the natural way to grow outward from the index
- **Configurable home note** — if no index-type note exists, user could designate one (or fall back to the most-connected node)

### What this does NOT require
- No new backend work (focus mode + topology already exist)
- No new data model changes (index type already defined)
- Minimal new UI (one toolbar button + auto-focus logic on load)
