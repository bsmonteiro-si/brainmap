Good news — **frontend logs already persist to disk** via the backend. Here's the current pipeline:

### Current Setup

1. **Frontend** calls `log.info(target, msg, fields)` → prints to browser console with `[brainmap]` prefix
2. **`forwardToBackend()`** detects Tauri context and invokes the `write_log` Tauri command
3. **Backend `write_log`** (`commands.rs:800`) re-emits via Rust `tracing` macros with `[frontend]` prefix
4. **Result**: frontend logs land in `~/.brainmap/logs/brainmap.log` (daily-rotated NDJSON) alongside backend logs

So frontend logs are already structured and saved to the same log files as backend logs. They're distinguishable by the `[frontend]` prefix in the message.

### What might you want to improve?

A few possible gaps I can see:

- **Separate frontend log file** — currently frontend and backend logs are mixed in the same file
- **Frontend-only log file** — a dedicated `frontend.log` for easier filtering
- **Log level filtering** — the `BRAINMAP_LOG` env var controls backend filtering, but frontend always forwards everything
- **Richer structured fields** — the fields are currently stringified JSON passed as a flat string, not parsed as structured tracing fields
- **Offline/batch buffering** — if the Tauri bridge fails, logs are lost (only console)

All four levels are forwarded — the frontend `forwardToBackend()` sends whatever level was logged, and `write_log` handles ERROR, WARN, DEBUG explicitly, with INFO as the default/fallback. So `log.debug()` does get sent to the file.

**However**, there's a filtering nuance: the Tauri backend default level is `info` (set in `lib.rs:20`). This means `debug!()` calls from `write_log` would be **filtered out by tracing** before reaching the file unless `BRAINMAP_LOG` is set to `debug` or lower.

So the current behavior is:
- `log.error()`, `log.warn()`, `log.info()` → forwarded AND written to file (level >= info)
- `log.debug()` → forwarded but **silently dropped** by tracing's level filter (default: info)
