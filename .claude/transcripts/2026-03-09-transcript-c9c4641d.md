## Assistant

There's a mix of both. Here's how I see it:

## What's sequential (has hard dependencies)

**1b must come before 1c, 2, and 3.** The CLI smoke commands are where you validate the core library's public API under real usage. If the API surface is wrong, you want to catch that before building a desktop app or MCP server on top of it. The roadmap explicitly calls this out as the exit criterion for 1b.

## What can be parallelized

Once 1b is done and the core API is stable:

**1c, 2, and 3 are largely independent.** They're all thin wrappers over the same core library:
- 1c (Desktop) — Tauri commands calling core functions, React UI
- 2 (Full CLI) — More clap commands calling core functions
- 3 (MCP) — MCP tool handlers calling core functions

These could be built in parallel since they don't depend on each other. The main risk is that one interface reveals an API gap that forces a core change, affecting the others — but 1b should catch most of those.

## Within 1b itself, there's a natural order

I'd suggest:

1. **Node CRUD** (`node create/read/update/delete/list`) — foundation for everything else
2. **Link CRUD** (`link create/list`) — depends on nodes existing
3. **Search** — independent of link CRUD, could go parallel with step 2
4. **Neighbors** — depends on both nodes and links being in place
5. **Output formatting** (`--format json/text`) — cut across all commands, best done last or as you go

## My recommendation

Build 1b sequentially (it's not that large), then fork into 1c + 2 + 3 in parallel. Within that parallel phase, I'd prioritize **3 (MCP server)** since it's the most differentiating feature and relatively small surface area, then **1c (desktop)** which carries the most risk (graph rendering is notoriously finicky, as your roadmap notes).
