### 6. Link/Note Autocomplete

**New file:** `crates/app/src/components/Editor/cmNoteAutocomplete.ts`
**New test file:** `crates/app/src/components/Editor/cmNoteAutocomplete.test.ts`

**Dependency:** `@codemirror/autocomplete` (already added in Phase A item 3).

**Implementation:**
- Import `autocompletion` from `@codemirror/autocomplete`.
- Create a custom `CompletionSource` that triggers on `[[` (wiki-link style) and `](` (standard link URL position).
- The completion source reads note paths from `useGraphStore.getState().nodes` (a `Map<string, NodeDto>` where `NodeDto = { path, title, note_type, tags }`).
- For `[[` trigger: show completions as `title` (with `detail` = path). On apply, insert the resolved relative path.
- For `](` trigger: show completions as paths. On apply, insert the path and closing `)`.

**Detailed approach:**
```ts
import { autocompletion, type CompletionContext, type CompletionResult } from "@codemirror/autocomplete";
import { useGraphStore } from "../../stores/graphStore";

function noteCompletionSource(context: CompletionContext): CompletionResult | null {
  // Check for [[ trigger
  const wikiMatch = context.matchBefore(/\[\[[^\]]*$/);
  if (wikiMatch) {
    const query = wikiMatch.text.slice(2).toLowerCase();
    const nodes = useGraphStore.getState().nodes;
    const options = Array.from(nodes.values())
      .filter(n => n.note_type !== "folder")
      .filter(n => n.title.toLowerCase().includes(query) || n.path.toLowerCase().includes(query))
      .slice(0, 20)
      .map(n => ({
        label: n.title,
        detail: n.path,
        apply: `[[${n.path}]]`,
      }));
    return { from: wikiMatch.from, options };
  }

  // Check for ]( trigger (standard markdown link URL)
  const linkMatch = context.matchBefore(/\]\([^)]*$/);
  if (linkMatch) {
    const urlStart = linkMatch.text.indexOf("(") + 1;
    const query = linkMatch.text.slice(urlStart).toLowerCase();
    const nodes = useGraphStore.getState().nodes;
    const options = Array.from(nodes.values())
      .filter(n => n.note_type !== "folder")
      .filter(n => n.path.toLowerCase().includes(query) || n.title.toLowerCase().includes(query))
      .slice(0, 20)
      .map(n => ({
        label: n.path,
        detail: n.title,
        apply: (view, completion, from, to) => {
          // Insert path + closing paren
          view.dispatch({ changes: { from: linkMatch.from + urlStart, to: context.pos, insert: n.path + ")" } });
        },
      }));
    return { from: linkMatch.from + urlStart, options };
  }

  return null;
}

export function noteAutocomplete(): Extension {
  return autocompletion({
    override: [noteCompletionSource],
    activateOnTyping: true,
  });
}
```

**Extension position:** Add `noteAutocomplete()` to the extensions array after `search()`. Note that `autocompletion` and `closeBrackets` both come from `@codemirror/autocomplete`. Using `autocompletion` with `override` means the default completion sources are replaced; only the custom `noteCompletionSource` is active. This is fine -- we do not want generic word completion.

**Important:** If `closeBrackets()` is also present, both can coexist. `closeBrackets` handles character insertion; `autocompletion` handles completion popup. No conflict.

**CSS:** Add styles for `.cm-tooltip-autocomplete` to match the app theme (dark/light). The autocomplete popup inherits from `.cm-tooltip`.

**Gotcha:** Calling `useGraphStore.getState()` inside a CM completion source is synchronous. The graph store is populated on workspace load, so nodes are always available. The completion is purely in-memory, no async needed.

**Testing (unit):**
- Export `noteCompletionSource` and test with a mock `CompletionContext`.
- Actually, since `CompletionContext` is hard to construct in tests, test the filtering/matching logic as a separate pure function: `filterNotes(nodes: NodeDto[], query: string): CompletionOption[]`.

export interface HeadingEntry {
  level: number;   // 1-6
  text: string;
  lineNumber: number;
}

export function extractHeadings(text: string): HeadingEntry[] {
  const result: HeadingEntry[] = [];
  const lines = text.split("\n");
  for (let i = 0; i < lines.length; i++) {
    const m = lines[i].match(/^(#{1,6})\s+(.+)/);
    if (m) {
      result.push({ level: m[1].length, text: m[2].trim(), lineNumber: i + 1 });
    }
  }
  return result;
}

### 9. Heading Folding

**New file:** `crates/app/src/components/Editor/cmHeadingFold.ts`
**New test file:** `crates/app/src/components/Editor/cmHeadingFold.test.ts`

**Dependencies:** `foldService`, `foldGutter`, `foldKeymap` from `@codemirror/language` (already installed).

**Implementation:**
- Create a custom `foldService` that defines fold ranges for headings. A heading's fold range extends from the end of the heading line to the line before the next heading of equal or higher level (or end of document).
- Optionally add `foldGutter()` for visual fold indicators in the gutter.
- Add `foldKeymap` to the keymap array (`Mod-Shift-[` to fold, `Mod-Shift-]` to unfold).

**Pure function:**
```ts
export function computeHeadingFoldRange(
  doc: Text,
  lineNumber: number,
): { from: number; to: number } | null {
  const line = doc.line(lineNumber);
  const m = line.text.match(/^(#{1,6})\s/);
  if (!m) return null;
  const level = m[1].length;

  // Find end: next heading of same or higher level, or end of doc
  let endLine = doc.lines;
  for (let i = lineNumber + 1; i <= doc.lines; i++) {
    const nextLine = doc.line(i);
    const nm = nextLine.text.match(/^(#{1,6})\s/);
    if (nm && nm[1].length <= level) {
      endLine = i - 1;
      break;
    }
  }

  if (endLine <= lineNumber) return null;
  return { from: line.to, to: doc.line(endLine).to };
}
```

**Extension position:** Add `foldGutter()` to the shared extensions array (before the readOnly branch), and `...foldKeymap` to the keymap array.

**Gotcha:** Must import `foldGutter` from `@codemirror/language` (not from `@codemirror/view`). Also, `foldGutter` adds gutter markers. If `lineNumbers()` is also present, the fold gutter appears next to line numbers. Ensure CSS does not clash.

**Interaction with `scanFencedBlocks`:** The fold service should NOT offer folds for headings inside fenced code blocks. Use the existing `scanFencedBlocks` utility.

**Testing:**
- `computeHeadingFoldRange` for `# A\ntext\n## B` at line 1 -> `{from: endOfLine1, to: endOfLine2}` (folds up to but not including `## B`)
- Heading at end of document -> fold to end of document
- Non-heading line -> null
