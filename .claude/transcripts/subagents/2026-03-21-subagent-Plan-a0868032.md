## Design Considerations

1. **Data model**: Shape variant stored as `shape` field in node data. Default is "rectangle" (backward compatible - undefined = rectangle).

2. **JSON Canvas compatibility**: The JSON Canvas spec doesn't have a `shape` field. We should store it as an extra field that won't break other tools reading the file. The translation layer already preserves unknown fields or we can explicitly handle it.

3. **Rendering approach**: Each shape is primarily CSS-driven. The same `CanvasTextNode` component handles all shapes, just applying a different CSS class based on `data.shape`. Some shapes (hexagon, speech bubble, sticky note fold) may need CSS clip-path or pseudo-elements.

4. **Shape picker**: Add a shape button to the `CanvasNodeToolbar` (alongside delete, border color, bg color). Could also split the "Add Card" button into a dropdown showing shape options.

5. **Creation flow**: Bottom toolbar "Add Card" could become a dropdown/popover showing shape thumbnails, or we keep the default and let users change shape via the node toolbar after creation.

## Implementation Plan: Customizable Text Card Shapes for Canvas Editor

### Overview

Add 5 shape variants for text cards (rounded, circle/oval, sticky note, callout/speech bubble, hexagon/diamond) with an extensible registry pattern so future shapes can be added with minimal code changes.

### 1. Shape Registry (new file: `canvasShapes.ts`)

Create a single source of truth for all shapes. This is the extensibility point -- adding a new shape later means adding one entry here.

```
crates/app/src/components/Editor/canvasShapes.ts
```

Define a `CanvasShape` type and a `CANVAS_SHAPES` array:

```ts
export type CanvasShapeId = "rectangle" | "rounded" | "circle" | "sticky" | "callout" | "hexagon";

export interface CanvasShapeDefinition {
  id: CanvasShapeId;
  label: string;
  icon: string;           // lucide icon name or inline SVG path
  cssClass: string;       // CSS modifier class applied to .canvas-text-node
  defaultWidth: number;   // suggested width when creating
  defaultHeight: number;  // suggested height when creating
}

export const CANVAS_SHAPES: CanvasShapeDefinition[] = [
  { id: "rectangle", label: "Rectangle", icon: "Square",    cssClass: "",                         defaultWidth: 250, defaultHeight: 100 },
  { id: "rounded",   label: "Rounded",   icon: "RectangleHorizontal", cssClass: "canvas-shape--rounded",   defaultWidth: 250, defaultHeight: 100 },
  { id: "circle",    label: "Circle",    icon: "Circle",    cssClass: "canvas-shape--circle",     defaultWidth: 160, defaultHeight: 160 },
  { id: "sticky",    label: "Sticky",    icon: "StickyNote", cssClass: "canvas-shape--sticky",    defaultWidth: 200, defaultHeight: 200 },
  { id: "callout",   label: "Callout",   icon: "MessageSquare", cssClass: "canvas-shape--callout", defaultWidth: 260, defaultHeight: 120 },
  { id: "hexagon",   label: "Diamond",   icon: "Diamond",   cssClass: "canvas-shape--hexagon",    defaultWidth: 160, defaultHeight: 160 },
];

export const DEFAULT_SHAPE: CanvasShapeId = "rectangle";

export function getShapeDefinition(id: string | undefined): CanvasShapeDefinition {
  return CANVAS_SHAPES.find(s => s.id === id) ?? CANVAS_SHAPES[0];
}
```

### 2. Data Model Changes

**In `canvasTranslation.ts`:**

- Add `shape?: string` to `JsonCanvasTextNode` interface (alongside `text`, `color`, `bgColor`).
- In `canvasToFlow()`, for text nodes, propagate `shape` into `data.shape` if present.
- In `flowToCanvas()`, for text nodes, write `data.shape` back to the JSON Canvas output if it is not `"rectangle"` (or undefined). This keeps the default case clean and backward-compatible.

The `shape` field is a custom extension of the JSON Canvas spec. Since JSON Canvas is a permissive format (unknown keys are ignored), this will not break interop with other tools like Obsidian.

**Backward compatibility:** If `shape` is missing or undefined, the node renders as `"rectangle"` -- the current default. No migration needed for existing `.canvas` files.

### 10. Edge Cases and Risks

- **clip-path shapes (diamond):** `clip-path` clips overflow including the resize handles and connection handles. Mitigation: ensure handles are rendered outside the clipped element, or use a wrapper. Since React Flow handles are absolutely positioned siblings, they should be outside the clip. Verify during implementation -- if handles get clipped, move the clip-path to an inner div rather than the outer `.canvas-text-node`.
- **Sticky note rotation:** The `transform: rotate(-1.5deg)` may interfere with React Flow's position calculations. Keep the rotation small (1-2 degrees) and test drag/resize behavior. If problematic, apply rotation to an inner wrapper instead.
- **Textarea in non-rectangular shapes:** Edit mode textarea will naturally be rectangular. For circle/diamond shapes, the text may look odd in edit mode. Accept this as reasonable -- the display mode uses overflow hidden; edit mode can temporarily show a standard rectangular textarea overlay.
- **Callout tail position:** The `::after` triangle sits below the node. This might overlap with connection handles or nodes below. Keep it small (8px) and ensure it doesn't interfere with click targets.
