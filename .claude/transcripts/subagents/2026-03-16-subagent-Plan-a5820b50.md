```rust
#[tauri::command]
pub fn delete_plain_file(
    app: tauri::AppHandle,
    state: State<'_, AppState>,
    path: String,
) -> Result<(), String> {
    let root = state.resolve_root(None)?;
    let abs_path = state.with_slot(&root, |slot| {
        handlers::validate_relative_path(&slot.workspace.root, &path)
    })?;
    state.register_expected_write(&root, abs_path);
    state.with_slot(&root, |slot| handlers::handle_delete_plain_file(&slot.workspace, &path))?;
    emit_files_changed_event(&app, &root, vec![], vec![path.clone()]);
    Ok(())
}
```

Note: uses `with_slot` (not `with_slot_mut`) since we are not modifying workspace state -- the file is not in `ws.notes`. Emits `files-changed` event with the path in `removed_files` so the file tree updates.

### Edge Cases

- **Binary files**: `delete_plain_file` uses `std::fs::remove_file` which works for any file type
- **Hidden files** (e.g., `.gitignore`): `list_workspace_files` already includes these; deletion should work the same
- **Files with special characters**: `validate_relative_path` handles normalization
- **Race condition**: File deleted between confirmation dialog open and confirm click -- `remove_file` will return an error, caught by the try/catch in `handleDeleteConfirm`
- **`.brainmap/` directory files**: `list_workspace_files` scans all files including `.brainmap/` -- should we exclude these? Currently `collect_files_recursive` does not filter them. This is a pre-existing concern, not introduced by this change.
