## Implementation Plan: macOS Dock Menu for BrainMap Segments

### Overview

Add a macOS dock right-click menu showing recent segments (workspaces), allowing users to switch segments directly from the dock icon. Since Tauri v2 (2.10.3) has no native dock menu API, we use Objective-C interop via the `objc2` crate family to set an `NSMenu` on the application dock tile.

### Architecture Decision: `objc2` vs `objc`/`cocoa`

**Recommendation: Use `objc2` ecosystem** (`objc2`, `objc2-foundation`, `objc2-app-kit`).

Rationale:
- The `objc` + `cocoa` crates are effectively unmaintained; `objc2` is the actively maintained successor
- `objc2` provides safe, typed wrappers around Objective-C classes (no raw `msg_send!` strings)
- Rust 1.94 is recent enough to support `objc2`'s latest features
- `objc2-app-kit` provides `NSMenu`, `NSMenuItem`, `NSApplication`, `NSDockTile` directly as typed structs

### Data Flow

```
Frontend (segmentStore mutation)
  --> invoke("update_dock_menu", { segments: [...] })
  --> Rust: update_dock_menu command
  --> dock_menu::set_dock_menu(segments, app_handle)
      --> Creates NSMenu with NSMenuItems
      --> Each item click --> app_handle.emit("brainmap://dock-open-segment", { path })
  --> Frontend listens for "brainmap://dock-open-segment"
      --> Routes to openWorkspace/switchSegment logic
```

#### Step 2: Create `dock_menu.rs` Module (macOS Only)

**New file: `crates/app/src-tauri/src/dock_menu.rs`**

This module is entirely behind `#[cfg(target_os = "macos")]`.

Key design decisions:

**2a. DockSegmentInfo struct:**
```rust
#[derive(serde::Deserialize, Clone)]
pub struct DockSegmentInfo {
    pub name: String,
    pub path: String,
}
```

**2b. Menu item click handling:**

The tricky part is getting from an Objective-C menu action callback back into Rust/Tauri. The approach:

1. Store the `AppHandle` in a global `OnceLock<AppHandle>` (set once during app setup).
2. For each `NSMenuItem`, use `setRepresentedObject:` to store the segment path as an `NSString`.
3. Create a custom Objective-C class (via `objc2::declare_class!`) that implements the `@selector(dockMenuItemClicked:)` action. This class reads the `representedObject` from the sender and calls `app_handle.emit("brainmap://dock-open-segment", payload)`.

**2c. Thread safety:**

`NSMenu` and `NSApplication` must be accessed from the main thread. Tauri commands run on the main thread by default for sync commands, but async commands use the async runtime. We need to use `dispatch::Queue::main().exec_async()` or `objc2`'s `MainThreadMarker` to ensure main-thread execution. Since Tauri v2's `AppHandle` has `run_on_main_thread()`, we can use that.

**2d. Implementation skeleton:**

```rust
use std::sync::OnceLock;
use tauri::{AppHandle, Emitter};

static APP_HANDLE: OnceLock<AppHandle> = OnceLock::new();

/// Call once during app setup to store the AppHandle.
pub fn init(app: &AppHandle) {
    let _ = APP_HANDLE.set(app.clone());
}

/// Rebuild the dock menu with the given segments.
/// Must be called from a context where main-thread dispatch is possible.
pub fn update(segments: Vec<DockSegmentInfo>) {
    // Use AppHandle::run_on_main_thread to ensure NSMenu ops happen on main thread
    if let Some(app) = APP_HANDLE.get() {
        let app = app.clone();
        let _ = app.run_on_main_thread(move || {
            unsafe { rebuild_menu(&segments) };
        });
    }
}
```

The `rebuild_menu` function:
1. Get `NSApplication::sharedApplication()`
2. Create a new `NSMenu`
3. For each segment (capped at 10), create an `NSMenuItem` with the segment name as title
4. Set the item's `action` to `sel!(dockMenuItemClicked:)` and `target` to the singleton handler object
5. Set `representedObject` to an `NSString` of the segment path
6. Add a separator after the segment list
7. Optionally add "Open Folder..." item that emits a different event
8. Call `dockTile().setMenu()` -- **correction**: the dock menu is provided via `NSApplicationDelegate.applicationDockMenu:`. Since Tauri already sets the NSApp delegate, we cannot override it.

**Alternative approach for setting the dock menu:** Instead of the delegate method, we can use the undocumented but widely-used approach of replacing the dock menu directly. However, the proper macOS API requires implementing `applicationDockMenu:` on the app delegate. Since Tauri owns the delegate, the cleaner approach is:

- Use `objc2` runtime to **add a method** to Tauri's existing delegate class via `class_addMethod`. This is a common pattern -- add `applicationDockMenu:` to the existing delegate class at runtime, returning our stored `NSMenu`.
- Alternatively, store the menu in a `static` and use method swizzling, but adding the method is cleaner since Tauri does not implement `applicationDockMenu:`.

**Revised approach:**

```rust
// At init time:
// 1. Get NSApp's delegate
// 2. Get its class
// 3. Add applicationDockMenu: method to that class (class_addMethod)
//    The method implementation returns our stored NSMenu
// 4. Store the NSMenu in a static Mutex<Option<Id<NSMenu>>>

// At update time:
// 1. Build new NSMenu with items
// 2. Replace the stored NSMenu
// (next time macOS calls applicationDockMenu:, it returns the updated menu)
```

This is safe because:
- `class_addMethod` only adds if the method doesn't exist (returns false if it does)
- Tauri v2 does NOT implement `applicationDockMenu:` on its delegate
- The method is added to the class once; the menu pointer is updated atomically

**2e. Menu item click handler:**

For the click handler, we define an extern "C" function that matches the Objective-C method signature:

```rust
extern "C" fn dock_item_clicked(this: &NSObject, _cmd: Sel, sender: &NSMenuItem) {
    // Get represented object (the segment path)
    let path_ns: &NSString = unsafe { sender.representedObject() }...;
    let path = path_ns.to_string();
    
    // Emit Tauri event
    if let Some(app) = APP_HANDLE.get() {
        let _ = app.emit("brainmap://dock-open-segment", DockOpenPayload { path });
    }
}
```

```typescript
async function syncDockMenu(segments: Segment[]) {
  const sorted = [...segments]
    .sort((a, b) => new Date(b.lastOpenedAt).getTime() - new Date(a.lastOpenedAt).getTime())
    .slice(0, 10);
```

#### Step 6: Frontend -- Listen for Dock Menu Events

**File: `crates/app/src/App.tsx`**

Add a `useEffect` that listens for the dock-open-segment event:

```typescript
useEffect(() => {
  let unlisten: (() => void) | null = null;
  let cancelled = false;
  
  listen<{ path: string }>("brainmap://dock-open-segment", async (e) => {
    const path = e.payload.path;
    const segStore = useSegmentStore.getState();
    const segment = segStore.getSegmentByPath(path);
    
    if (!segment) return; // Segment was removed since menu was built
    
    const wsStore = useWorkspaceStore.getState();
    
    if (!wsStore.info) {
      // On home screen -- open the segment
      await wsStore.openWorkspace(path);
      const state = useWorkspaceStore.getState();
      if (state.info) {
        segStore.touchSegment(segment.id);
        segStore.addOpenSegment(segment.id);
        segStore.setActiveSegmentId(segment.id);
      }
    } else if (segStore.activeSegmentId !== segment.id) {
      // Inside a different segment -- switch to it
      segStore.addOpenSegment(segment.id);
      await wsStore.switchSegment(segment.id);
    }
    // If already the active segment, no-op (just bring window to front, which macOS does automatically)
  }).then((fn) => {
    if (cancelled) fn();
    else unlisten = fn;
  });
  
  return () => {
    cancelled = true;
    unlisten?.();
  };
}, []);
```

### Edge Cases

1. **No segments**: The dock menu will be empty (just the standard macOS items like "Options" remain). This is fine -- the custom menu simply has no items.

2. **Very long segment list**: Capped at 10 most recent in the `syncDockMenu` function.

3. **Segment removed while menu shows it**: The event listener checks `getSegmentByPath` and silently ignores if not found.

4. **Rapid segment mutations**: Each call to `update` rebuilds the menu from scratch. This is cheap (a few NSMenuItem allocations) and idempotent.

5. **App not yet fully initialized**: `APP_HANDLE.get()` returns `None` before `setup()` runs, so `update()` is a safe no-op.

6. **Concurrent switchSegment**: The existing `switchInProgress` guard in `workspaceStore` protects against double-switching.

7. **Memory management**: `objc2`'s `Id<NSMenu>` and `Id<NSMenuItem>` are reference-counted (`retain`/`release`). Storing the menu in a `static Mutex<Option<Id<NSMenu>>>` ensures the old menu is released when replaced.
