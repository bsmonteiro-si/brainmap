## Implementation Plan: Comprehensive Code Block Syntax Highlighting

### Problem Analysis

Both `oneDarkHighlightStyle` and `defaultHighlightStyle` do cover code tokens, but they have key gaps:

1. **`oneDarkHighlightStyle`** groups `tags.name`, `tags.propertyName`, `tags.character`, and `tags.deleted` into the same coral color (`#e06c75`). It also lumps `tags.typeName`, `tags.className`, `tags.number`, `tags.annotation`, `tags.modifier` all into chalky (`#e5c07b`). This means JSON property names and string values can end up the same color, and numbers look the same as type names.

2. **`defaultHighlightStyle`** is even sparser -- it has no `propertyName` rule at all (only `definition(propertyName)`), and its color choices (`#708`, `#219`, `#a11`) produce a washed-out look on both light and dark backgrounds.

3. Neither style has `fontStyle: "italic"` on comments, which is a hallmark of premium code themes.

4. The real issue for JSON: the JSON Lezer parser emits `tags.propertyName` for keys and `tags.string` for string values. In `oneDarkHighlightStyle`, `tags.propertyName` is coral and `tags.string` (via `tags.inserted`) is sage -- these should work. But the `tags.name` rule at line 95-96 also matches `propertyName` and `name` simultaneously, and because CM6 picks the first matching rule, it may not be clear. The actual problem is likely that the bundled `oneDarkHighlightStyle` does not provide enough differentiation between JSON-specific tokens like `tags.null` vs `tags.bool` vs `tags.number` -- they all end up the same whiskey/chalky color.

### Why This Works

The root cause of poor JSON highlighting: the JSON Lezer grammar emits these specific tags:
- `tags.propertyName` for object keys (e.g., `"name"`)
- `tags.string` for string values
- `tags.number` for numeric values
- `tags.bool` for `true`/`false`
- `tags.null` for `null`
- `tags.punctuation` for `{`, `}`, `[`, `]`, `:`, `,`

In the current `oneDarkHighlightStyle`, `tags.propertyName` and `tags.name` share the same coral color, which is correct. But `tags.string` maps to sage, and `tags.number`/`tags.bool` map to chalky/whiskey -- these are distinguishable but the bundled `oneDarkHighlightStyle` groups too many tags together which can cause conflicts when both markdown and code tags overlap in CM6's resolver. By providing a comprehensive, explicit mapping for every token type, the new style eliminates ambiguity and ensures every token gets its intended color.

For Java, the Lezer grammar emits `tags.keyword`, `tags.typeName`, `tags.function(tags.variableName)`, `tags.string`, `tags.comment`, `tags.className`, `tags.operator`, etc. The new style gives each a distinct, purposeful color that follows the One Dark Pro / IntelliJ Darcula conventions developers expect.
