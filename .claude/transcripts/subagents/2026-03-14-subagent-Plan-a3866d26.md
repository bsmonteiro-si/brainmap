## Implementation Plan: Per-Component Themes (Files Panel + Editor Panel)

### Overview

Add two new theme selectors in the Settings modal that allow the Files panel (and left content panel including search) and the Editor panel to each have an independent theme override. The approach leverages the existing CSS `[data-theme]` attribute mechanism, placing `data-theme` attributes on panel container `div` elements so CSS custom properties cascade correctly within each subtree.

### Key Insight: CSS `[data-theme]` Cascading

The current `[data-theme="light"]` and `[data-theme="dark"]` CSS blocks define CSS custom variables (`--bg-primary`, `--text-primary`, etc.). When a `data-theme` attribute is placed on a child element, CSS variable definitions from the matching `[data-theme]` selector are scoped to that subtree. Any descendant using `var(--bg-primary)` will resolve to the overridden value. This means **no new CSS variable definitions are needed** -- the existing theme blocks already handle it.

The only selectors that use `[data-theme] .descendant` patterns (not just variable definitions) are all graph-related: `.graph-toolbar`, `.graph-legend`, `.graph-node-tooltip`, `.edge-filter-popover`, `.graph-minimap`, `.related-note-tooltip`. These all live inside the content panel, so when a `data-theme` attribute is placed on the content panel container, those selectors will match correctly because the `[data-theme]` is an ancestor of those elements. No CSS changes needed for these selectors either.

### Concern: CodeMirror Theme

The `MarkdownEditor` component reads `effectiveTheme` from the global UI store to configure syntax highlighting (`oneDark` extension, `buildMarkdownHighlight(isDark)`). With per-editor theming, it must read the **editor panel's effective theme** instead. This requires either:
- A new derived value in the store (`effectiveEditorTheme`), or
- A `resolveComponentTheme` helper the component calls.

The derived value approach is cleaner since it avoids duplicating resolution logic in components.

Add a resolution helper:
```typescript
function resolveComponentTheme(component: ComponentTheme, global: "light" | "dark"): "light" | "dark" {
  return component === "inherit" ? global : component;

```

Initialize from stored prefs (defaults to `"inherit"`):
```typescript

editorTheme: storedPrefs.editorTheme ?? "inherit",
effectiveFilesTheme: resolveComponentTheme(storedPrefs.filesTheme ?? "inherit", resolveTheme(storedPrefs.theme ?? "system")),
effectiveEditorTheme: resolveComponentTheme(storedPrefs.editorTheme ?? "inherit", resolveTheme(storedPrefs.theme ?? "system")),
```

Implement the setters, following the existing `savePrefs` pattern. Each setter updates both the raw preference and the derived effective value.

**Critical:** The existing `setTheme` action must also re-derive `effectiveFilesTheme` and `effectiveEditorTheme` when the global theme changes, because "inherit" components depend on the global effective theme. Update `setTheme` to:
```typescript
setTheme: (theme: Theme) => {
  const effective = resolveTheme(theme);
  const s = get();
  set({
    theme,
    effectiveTheme: effective,
    effectiveFilesTheme: resolveComponentTheme(s.filesTheme, effective),
    effectiveEditorTheme: resolveComponentTheme(s.editorTheme, effective),
  });
  savePrefs({ ... });
},
```

Also wire the system media query listener (if one exists) or the existing `resolveTheme` path to re-derive component themes when the system theme changes. Currently, `resolveTheme` is called at init and in `setTheme`. If the user has `theme: "system"`, changes to the OS theme preference are only detected on reload or manual toggle. This is existing behavior and does not need to change for this feature -- but worth noting.
