## Implementation Plan: Untitled Tab Support

### Design Decisions

**1. Tab ID scheme**

Untitled tabs use a synthetic ID with the prefix `__untitled__/` followed by a monotonically increasing counter: `__untitled__/1`, `__untitled__/2`, etc. This prefix cannot collide with real file paths because:
- Real paths are workspace-relative and never start with double underscores
- The `/` after `__untitled__` makes it structurally valid as a tab ID but impossible as a filename

The counter lives in `tabStore` as `_untitledCounter: number` (initialized to 0, incremented on each creation). It never resets during a session -- if you create untitled-1, close it, create another, it becomes untitled-2.

A helper function `isUntitledTab(id: string): boolean` checks the prefix. Export this from `tabStore.ts` so all consumers can use it.

**2. Tab kind: introduce "untitled"**

Extend the `kind` union from `"note" | "plain-file"` to `"note" | "plain-file" | "untitled"`. This is cleaner than trying to shoehorn untitled tabs into the note kind, because untitled tabs have no backing `NoteDetail` until saved, and the editor/auto-save logic must treat them differently.

The `TabState` type gains one new optional field:
```typescript
untitledBody: string | null;  // the buffer content for unsaved untitled tabs
```

Actually, the existing `editedBody` field already serves this purpose. When restoring an untitled tab, `editedBody` is the authoritative content (there is no `activeNote.body` fallback). So no new field is needed on `TabState`.

**3. EditorPanel rendering for untitled tabs**

When `activeTabId` points to an untitled tab, both `activeNote` and `activePlainFile` are `null`. Today this shows "Loading note..." or the empty state. We need a third rendering path.

Add a new state field to `editorStore`:
```typescript
isUntitledTab: boolean;  // true when the active tab is untitled
```

When `isUntitledTab` is true, EditorPanel renders:
- TabBar (always)
- A simplified hero header showing the untitled title (e.g., "Untitled-1") with a neutral type pill
- Edit/Preview toggle (same as existing)
- EditorToolbar (same as existing)
- MarkdownEditor with `content={editedBody ?? ""}` and `notePath={activeTabId}`
- No FrontmatterForm, no RelatedNotesFooter (there are no links yet)

**4. Tab switching: `openUntitledTab` and `activateUntitledTab`**

Add to `editorStore`:

```typescript
openUntitledTab: () => void;        // creates a new untitled tab and activates it
activateUntitledTab: (id: string) => void;  // switches to an existing untitled tab
```

`openUntitledTab`:
1. Snapshot current tab state (`snapshotToActiveTab()`)
2. Auto-save current tab if dirty (unless current tab is also untitled)
3. Increment `tabStore._untitledCounter`, generate ID
4. Call `tabStore.openTab(id, "untitled", "Untitled-N", null)`
5. Set editor state: `{ ...CLEAN_EDITOR_STATE, isUntitledTab: true, isLoading: false, viewMode: "edit" }`

`activateUntitledTab(id)`:
1. Snapshot current tab state
2. Auto-save current tab if dirty and not untitled
3. Activate tab in tabStore
4. Restore from tab snapshot: `{ editedBody: tab.editedBody, isDirty: tab.isDirty, isUntitledTab: true, activeNote: null, activePlainFile: null, ... }`

The existing `openNote`/`openPlainFile` need no changes -- they already set `isUntitledTab: false` implicitly (we add it to `CLEAN_EDITOR_STATE`).

**5. Save As flow: reuse CreateNoteDialog**

When the user presses Cmd+S on an untitled tab, instead of calling `saveNote()` directly, we open the CreateNoteDialog with the untitled tab's body pre-populated. This is the "Save As" flow.

Approach:
- Add `saveAsMode` to uiStore's create note dialog state:
  ```typescript
  createNoteSaveAsBody: string | null;   // pre-fill body
  createNoteSaveAsTabId: string | null;  // which untitled tab to replace
  ```
- When CreateNoteDialog detects `saveAsMode`, it:
  - Pre-fills the body field (and makes it read-only or hidden, since user already wrote it)
  - On successful create, closes the untitled tab and opens the new note tab
  - The untitled tab's `editedBody` becomes the body of the new note

The Cmd+S handler in App.tsx changes:
```typescript
if (isMod && e.key === "s") {
  e.preventDefault();
  const tabId = useTabStore.getState().activeTabId;
  if (tabId && isUntitledTab(tabId)) {
    // Open Save As dialog
    const tab = useTabStore.getState().getTab(tabId);
    useUIStore.getState().openCreateNoteDialog({
      mode: "save-as",
      saveAsBody: editorStore.editedBody ?? "",
      saveAsTabId: tabId,
    });
  } else {
    useEditorStore.getState().saveNote();
  }
}
```

**6. Close confirmation: `UnsavedChangesDialog` component**

A new component `/components/Layout/UnsavedChangesDialog.tsx` with three buttons: Save / Don't Save / Cancel.

State management in uiStore:
```typescript
unsavedChangesDialogOpen: boolean;
unsavedChangesTabId: string | null;
unsavedChangesCallback: ((action: "save" | "discard" | "cancel") => void) | null;
```

Wait -- storing callbacks in Zustand state is an anti-pattern. Instead, use a promise-based approach:

Create a module `stores/unsavedChangesPrompt.ts`:
```typescript
let resolvePrompt: ((action: "save" | "discard" | "cancel") => void) | null = null;

export function promptUnsavedChanges(tabId: string): Promise<"save" | "discard" | "cancel"> {
  useUIStore.getState().openUnsavedChangesDialog(tabId);
  return new Promise((resolve) => {
    resolvePrompt = resolve;
  });
}

export function resolveUnsavedChanges(action: "save" | "discard" | "cancel") {
  useUIStore.getState().closeUnsavedChangesDialog();
  resolvePrompt?.(action);
  resolvePrompt = null;
}
```

The dialog component calls `resolveUnsavedChanges(action)` on button click.

UIStore additions:
```typescript
unsavedChangesDialogOpen: boolean;
unsavedChangesTabId: string | null;
openUnsavedChangesDialog: (tabId: string) => void;
closeUnsavedChangesDialog: () => void;
```

Usage in close flows:
```typescript
// In handleClose for untitled dirty tabs:
const action = await promptUnsavedChanges(tabId);
if (action === "save") {
  // Open Save As dialog, wait for completion, then close tab
} else if (action === "discard") {
  closeTabAndNavigateNext(tabId);
} else {
  // cancel -- do nothing
}
```

For non-untitled dirty tabs, the existing auto-save-on-close behavior remains unchanged.

**7. Window close interception**

Tauri v2 supports window close interception via the `close_requested` event. In `App.tsx`, listen for this event:

```typescript
import { getCurrentWindow } from "@tauri-apps/api/window";

useEffect(() => {
  const appWindow = getCurrentWindow();
  const unlisten = appWindow.onCloseRequested(async (event) => {
    // Check for any dirty untitled tabs
    const dirtyUntitled = useTabStore.getState().tabs.filter(
      (t) => t.kind === "untitled" && t.isDirty
    );
    if (dirtyUntitled.length > 0) {
      event.preventDefault();
      // Show dialog for each dirty untitled tab (or a batch dialog)
      for (const tab of dirtyUntitled) {
        const action = await promptUnsavedChanges(tab.id);
        if (action === "cancel") return; // abort close
        if (action === "discard") continue;
        if (action === "save") {
          // Need to handle save-as flow... this gets complex
        }
      }
      // All resolved -- close the window
      appWindow.close();
    }
  });
  return () => { unlisten.then((f) => f()); };
}, []);
```

For simplicity, if there are multiple dirty untitled tabs on window close, show a single batch confirmation: "You have N unsaved untitled notes. Save all / Discard all / Cancel". The Save All path would sequentially open CreateNoteDialog for each. This is simpler but might be annoying with many untitled tabs -- a reasonable V1 simplification.

Alternative V1: on window close with dirty untitled tabs, show a single dialog: "You have unsaved changes in N untitled tab(s). Discard all / Cancel". Skip the Save option on window close to avoid the complex sequential Save-As UI. Users can cancel and save individually. This is much simpler and matches how some editors handle this.

**8. Counter state**

Lives in `tabStore` as `_untitledCounter: number`. Starts at 0, increments each time `openUntitledTab` is called. Never decrements. Resets to 0 on `tabStore.reset()` (workspace close).

**9. Cmd+N behavior change**

Change Cmd+N to create an untitled tab instead of opening CreateNoteDialog. This is the more natural VS Code-style behavior. The CreateNoteDialog remains accessible from:
- File tree toolbar "+" button
- File tree right-click context menu
- Save As flow (Cmd+S on untitled tab)

**10. Auto-save interaction**

The `useAutoSave` hook must skip untitled tabs. In `trySave()`:
```typescript
const activeTabId = useTabStore.getState().activeTabId;
if (activeTabId && isUntitledTab(activeTabId)) return;
```

### Edge Cases

1. **Multiple untitled tabs**: Fully supported. Each gets a unique ID and counter.
2. **Switching from untitled to real tab**: Snapshot untitled tab content, auto-save real tab on switch-back.
3. **Switching from real to untitled**: Auto-save real tab, restore untitled tab content from snapshot.
4. **Save As with duplicate path**: CreateNoteDialog already validates this (API returns error).
5. **Save As canceled**: Tab remains untitled with content preserved.
6. **Workspace close with untitled tabs**: `tabStore.reset()` clears everything. The workspace close flow already calls reset. If there are dirty untitled tabs, the close-workspace action should check first (same pattern as window close).
7. **Undo store**: Creating a note via Save-As should push to undo store (already handled by CreateNoteDialog).
8. **File watcher events**: Untitled tabs have no backing file, so watcher events never match them.
9. **Navigation history**: Do not push untitled tab IDs to `navigationStore` (they are ephemeral and cannot be re-opened from history).
10. **Empty untitled tabs on close**: If an untitled tab has no content (editedBody is null or empty string), close it silently without prompting -- it has no user content worth saving.
