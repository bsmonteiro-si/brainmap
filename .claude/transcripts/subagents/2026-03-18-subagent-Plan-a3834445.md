## Review of the Proposed Fix

### 1. Is comparing parent directories the right heuristic?

Yes, `old_rp.parent() == new_rp.parent()` is the correct heuristic. It cleanly distinguishes:
- **Pure rename** (same parent): `Concepts/Foo.md` to `Concepts/Bar.md` -- title should update
- **Move to different folder**: `Concepts/Foo.md` to `Archive/Foo.md` -- title should NOT update
- **Move + rename** (different parent AND different stem): `Concepts/Foo.md` to `Archive/Bar.md` -- this is a move, not an inline rename from the file tree, so title should NOT update

The frontend `executeRenameItem` always produces paths with the same parent (it calls `computeRenamePath` which only replaces the last path segment). Drag-and-drop moves go through a different code path that changes the parent directory. So the heuristic aligns perfectly with how these operations are triggered.

One subtlety: `RelativePath::parent()` returns `Option<RelativePath>`. For root-level files (e.g., `foo.md`), `parent()` returns `None`. Two root-level files would both return `None`, so `None == None` is `true` -- this is correct behavior; renaming a root-level file should update its title.

### 2. Are there other places that need updating?

The proposed insertion point (after line 914, before line 915) is correct. The code flow is:

1. Lines 911-913: Rewrite outgoing link targets
2. **INSERT HERE**: Update title if same-parent rename
3. Line 915: `note.path = new_rp.clone()`
4. Line 917: `parser::serialize_note(&note)` -- writes frontmatter to disk (needs updated title)
5. Line 924: `self.graph.add_node(...)` -- reads `note.frontmatter.title.clone()` (needs updated title)
6. Line 934: `self.index.add_note(&note, mtime)` -- FTS indexes title (needs updated title)

All three consumers (serialization, graph, FTS index) are downstream of the insertion point. No other places need updating.

### 3. Should the frontend rename input show `node.title` instead of `node.name`?

**No, `node.name` (filename stem) is correct for the inline rename input.** The rename operation changes the filename, not the title directly. Showing `node.name` lets the user edit exactly what will change on disk. The backend fix then syncs the title to match.

However, there is a UX inconsistency worth noting: the tree **displays** `node.title` (frontmatter title) but the rename input shows `node.name` (filename stem). In the seed dataset these are always identical, but if a user manually edits the frontmatter title to differ from the filename, the inline rename input would show a different string than what was displayed. This is an edge case but not a regression from this fix -- it is a pre-existing issue.

### 4. What about `move_folder` -- same issue?

`move_folder` (line 1006) does NOT update titles of notes inside the folder. However, this is **not a bug** for the same-parent rename case. When you rename a folder (e.g., `Concepts/` to `Topics/`), the notes inside keep their filenames (e.g., `Foo.md` stays `Foo.md`), so their titles should remain unchanged. The folder rename only changes the directory name, not any note filenames.

`move_folder` does NOT need a title update.

### Additional edge case: the proposed code snippet

The proposed code uses `Path::new(new_rp.as_str()).file_stem()`. This is correct -- `RelativePath` stores a string like `Concepts/New Name.md`, and `Path::file_stem()` returns `"New Name"`. The `unwrap_or_else` fallback to `new_path.to_string()` would only trigger if the path has no file stem (e.g., just `.md`), which is blocked by frontend validation (`validateRenameNameFormat` rejects names starting with `.`).

One minor suggestion: consider using `new_rp.as_str()` instead of `new_path` in the fallback, since `new_path` is the raw string parameter while `new_rp` is the parsed `RelativePath`. Both should be equivalent but `new_rp` is more canonical. This is a non-issue in practice since the fallback path is unreachable.
